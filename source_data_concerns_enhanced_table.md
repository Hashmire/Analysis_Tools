## **üîß Architectural Overview: Update Pattern Preprocessing**

**CRITICAL ARCHITECTURE CHANGE**: All source data concern checks now use **curated platform data** that has been preprocessed through update pattern transformation. This prevents false positives from legitimate update patterns like "3.3 patch1" being flagged as invalid characters.

**Preprocessing Flow:**
1. **Raw Platform Data**: Original CVE source data as received
2. **Update Pattern Processing**: `transform_version_with_update_pattern()` applied to all version fields
3. **Curated Platform Data**: Clean base versions (e.g., "3.3 patch1" ‚Üí "3.3") used for analysis
4. **Strategic Usage**: 
   - **Raw data** for non-version fields (vendor, product, platforms)  
   - **Curated data** for all version-related checks (granularity, overlaps, character validation)

This architectural approach ensures that **legitimate update patterns are not flagged as data quality issues** while maintaining comprehensive validation of data concerns.
